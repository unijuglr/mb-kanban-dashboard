## Standard Proof Protocol for MB-024 / MB-025

A run only counts if all of the following are preserved.

### Required run inputs
1. exact invocation command
2. exact model name
3. exact prompt / task brief
4. target working directory
5. start and stop timestamps

### Required produced artifacts
1. at least one source file requested by the prompt
2. at least one execution output file generated by running the requested code
3. one machine-readable manifest file describing outputs and hashes
4. one human-readable run note summarizing result and failure mode if any

### Required validation checks
1. confirm target files exist
2. run the produced code with a bounded command
3. confirm stdout/stderr or saved output matches expected contract
4. record file hashes so later drift is detectable
5. distinguish clearly between:
   - successful model-mediated generation
   - deterministic fallback
   - manual human repair
   - total failure / hang

### Reliability bar
Call the local coder “reliable enough for backlog execution” only after:
- **3 consecutive successful model-mediated runs**
- on **at least 2 different bounded tasks**
- with saved artifacts and validation for each run
- with no silent hangs or empty-output false positives

Anything short of that is anecdote, not proof.

## Proposed Minimal Artifact Contract

For each future proof run, save under:

`artifacts/local-coder-runs/<run-id>/`

Contents:
- `request.txt` — exact prompt/task
- `command.txt` — exact invocation
- `agent.log` — captured stdout/stderr
- `manifest.json` — produced files, sha256, timestamps, exit codes
- `validation.txt` — bounded validation command outputs
- `result.md` — one-page human summary
- `workspace/` — generated working files if they should be retained

## File-Backed Pipeline Design for MB-026

Use the filesystem as the source of truth.

### Queue layout

```text
motherbrain-pipeline/
  inbox/
  claimed/
  qa/
  integrate/
  done/
  failed/
  logs/
  templates/
```

### Task file schema (v1)

```json
{
  "id": "mb-026-example-001",
  "created_at": "2026-04-03T21:00:00Z",
  "project": "mb-kanban-dashboard",
  "card": "MB-026",
  "stage": "coder",
  "status": "queued",
  "prompt_file": "templates/mb-026-example-001-prompt.md",
  "working_dir": "/absolute/path/to/repo",
  "expected_outputs": [
    "src/example.js",
    "PROOF_EXAMPLE.md"
  ],
  "validation": {
    "command": "npm test -- --runInBand",
    "timeout_seconds": 120
  },
  "upstream": [],
  "handoff": {
    "qa_required": true,
    "devops_required": false
  },
  "attempt": 0,
  "max_attempts": 2,
  "artifacts_dir": "logs/mb-026-example-001"
}
```

### Stage rules
- `inbox/`: planner-created queued work
- `claimed/`: one runner has exclusive ownership
- `qa/`: coder claims passed basic self-check and awaits QA runner
- `integrate/`: QA passed and integration/devops work remains
- `done/`: validated and merged/recorded
- `failed/`: exhausted retries or hit blocking defect

### Why this is better than the current embedded-only path
- hanging agents do not erase the task record
- each handoff becomes inspectable and replayable
- validation commands are explicit, bounded, and stored beside outputs
- trust comes from files on disk, not vibes in logs
